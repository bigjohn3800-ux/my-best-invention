# 마이베스트 발명창업 사이트 검증 및 수정·보완 의견서

작성일: 2026-06-17  
대상 프로젝트: `Invention-Ventures-Hub`  
로컬 미리보기 URL: `http://127.0.0.1:5173`  
검증 범위: 홈, 패키지/가격, 주요 SPA 라우트, 빌드 산출물, 서버 실행 전제, 상용화 준비도

## 1. 종합 의견

마이베스트 발명창업 사이트는 단순 교육 소개형 페이지에서 “발명 아이디어를 사업화 가능한 프로젝트로 전환하는 AI 워크스페이스” 방향으로 전환된 상태다. 홈 화면에는 발명 프로젝트 시작, 패키지 보기, 사업화 준비도, 비공개 프로젝트, 특허/IP, 시장 검증, BMC/IR/멘토 리뷰 흐름이 반영되어 있어 유료 전환을 위한 제품 메시지는 이전보다 훨씬 명확해졌다.

다만 현재 상태는 “상용화 컨셉과 정적 화면 검증 가능” 단계이며, 실제 사용자가 결제하거나 기관에 도입할 수 있는 운영형 서비스로 보려면 서버 타입 오류, 데이터베이스 연결, 인증/권한, 결제 활성화, 실제 브라우저 QA 자동화가 추가로 정리되어야 한다.

판정:

- 디자인/메시지: 상용화 방향으로 개선됨
- 정적 라우트 응답: 정상
- 클라이언트 빌드 산출물: 존재
- 실제 API/DB 기반 서비스 실행: 미완료
- 전체 타입 검증: 실패
- 결제 가능한 상품화 상태: 준비 전 단계

## 2. 검증 근거

### 2.1 로컬 파일 및 프로젝트 상태

현재 프로젝트 폴더에는 Replit ZIP에서 가져온 실제 앱 소스가 병합되어 있다.

확인된 주요 소스:

- `client/src/pages/home-page.tsx`
- `client/src/pages/pricing-page.tsx`
- `client/src/App.tsx`
- `client/src/components/navbar.tsx`
- `server/`
- `shared/`
- `docs/commercialization/`
- `supabase/`

현재 폴더는 Git 저장소가 아니므로 `git status` 기준 변경 이력 검증은 불가능하다. 추후 Git 저장소 초기화 또는 GitHub 연결이 필요하다.

### 2.2 라우트 응답 검증

정적 미리보기 서버 기준으로 아래 라우트가 모두 HTTP 200으로 응답했다.

| 경로 | 응답 | 비고 |
| --- | --- | --- |
| `/` | 200 | SPA 루트 정상 |
| `/pricing` | 200 | 패키지 페이지 라우트 정상 |
| `/invention-studio` | 200 | SPA 진입 정상 |
| `/startup-lab` | 200 | SPA 진입 정상 |
| `/dashboard` | 200 | SPA 진입 정상 |

주의: 정적 서버 검증은 React 앱 진입 HTML 응답을 확인한 것이다. API, 로그인, DB 저장, AI 생성, 관리자 권한까지 검증한 것은 아니다.

### 2.3 구현 반영 확인

홈 화면에는 다음 상용화 메시지가 코드에 반영되어 있다.

## 3. 현재 강점

### 3.1 제품 포지셔닝이 선명해짐

기존 교육형 문구 중심에서 벗어나 발명가와 예비창업자가 돈을 낼 만한 가치인 “결과물” 중심으로 메시지가 재구성되었다.

좋은 방향:

- 발명 아이디어를 프로젝트로 저장
- 특허/IP 준비도 강조
- 시장 검증과 BMC/IR 연결
- 멘토 리뷰와 기관 운영까지 확장
- 개인, 팀, 기관 패키지 구분

### 3.2 유료 전환 구조가 생김

`/pricing` 페이지를 통해 무료 체험, 실전 창업 준비팀, 기관 도입이라는 구매자 구분이 만들어졌다. 결제 기능이 아직 꺼져 있어도, 영업/상담/기관 제안에 사용할 수 있는 기본 상품 언어가 생겼다.

### 3.3 핵심 기능 라우트가 이미 존재함

발명 스튜디오, 창업 랩, 진단, 대시보드, 관리자 페이지가 라우터에 존재한다. 완전히 새로 만드는 상태가 아니라, 현재 구조를 상용화 기준으로 정리하면 된다.

## 4. 주요 문제점

### 4.1 실제 서버 실행에는 `DATABASE_URL`이 필요함

`server/db.ts`는 `DATABASE_URL`이 없으면 서버가 바로 중단되도록 되어 있다.

현재 `.env.example`에는 Supabase URL/Anon Key/Toss 키 자리는 있지만 `DATABASE_URL` 항목은 없다.

영향:

- Express 개발 서버 실행 불가
- 로그인/회원/프로젝트 저장 검증 불가
- 관리자 페이지 실제 데이터 검증 불가
- AI 생성 기록 저장 검증 불가

의견:

`DATABASE_URL`을 `.env.example`에 명시하고, Supabase Postgres 연결 문자열 또는 별도 Postgres 연결값을 로컬/스테이징/운영 환경별로 관리해야 한다.

### 4.2 TypeScript 검증 실패

`npm run check` 실행 결과 기존 서버 코드에서 타입 오류가 다수 확인되었다.

주요 유형:

- `string | string[]` 값을 `string` 인자로 넘기는 문제
- `p-retry`의 `AbortError` 타입 사용 문제
- 이미지 응답의 `response.data` undefined 가능성
- 세션 객체를 `Record<string, unknown>`으로 직접 변환하는 문제
- 진단 결과 타입을 `Record<string, string>`으로 변환하는 문제

영향:

- 배포 전 신뢰도 하락
- CI/CD에서 타입 검사 실패 가능
- 서버 코드 수정 시 회귀 위험 증가

의견:

상용화 전 반드시 서버 타입 오류를 정리해야 한다. 특히 외부 AI/Replit 연동 라우트와 주요 API 입력값은 런타임 검증까지 같이 넣는 것이 좋다.

### 4.3 실제 브라우저 시각 QA는 미완료

Playwright 패키지는 설치되어 있으나 Chromium headless shell 바이너리가 완전히 설치되지 않아 자동 스크린샷 검증은 완료하지 못했다.

확인된 한계:

- 정적 서버 응답은 검증됨
- 실제 브라우저 렌더 스크린샷은 미완료
- 모바일/데스크톱 시각 겹침 검증은 미완료

의견:

Playwright 브라우저 설치를 완료한 뒤 아래 화면을 최소 2개 뷰포트로 캡처해야 한다.

- 홈 `/`
- 패키지 `/pricing`
- 발명 스튜디오 `/invention-studio`
- 창업 랩 `/startup-lab`
- 로그인 `/auth`
- 대시보드 `/dashboard`
- 관리자 `/admin`

### 4.4 전체 빌드가 서버 번들 단계에서 제한됨

클라이언트 빌드는 Vite `runner` config loader 방식으로 통과했다. 다만 전체 `npm run build`는 서버 esbuild 번들 단계에서 로컬 샌드박스 접근 문제와 경로 해석 문제가 남아 있다.

영향:

- 정적 프론트 검증은 가능
- 서버 포함 프로덕션 번들 검증은 미완료
- 배포 전 빌드 파이프라인 보완 필요

의견:

서버 번들러 설정을 Windows/샌드박스 환경에서도 안정적으로 동작하게 정리하거나, Vercel/Node 배포 방식에 맞춘 별도 빌드 명령을 분리하는 것이 좋다.

### 4.5 결제 기능은 아직 “보여주기” 단계

패키지 페이지는 유료 전환을 위한 상품 구조를 보여주지만, 실제 결제 활성화는 아직 되어 있지 않다.

필수 결정:

- 실제 가격
- 무료 체험 범위
- 환불 정책
- 기관 견적 방식
- Toss Payments 결제 활성화 시점
- 결제 실패/취소/환불 처리
- 유료 권한 부여 기준

의견:

현재는 “유료 전환 준비”로 표현한 것이 적절하다. 바로 결제를 열기보다 문의/상담 기반으로 수요를 확인한 뒤 결제를 붙이는 순서가 안전하다.

## 5. 디자인/UX 보완 의견

### 5.1 홈 첫 화면

현재 방향은 좋다. 다만 실제 유료 전환을 높이려면 첫 화면에서 아래 3가지를 더 명확히 보여야 한다.

보완 제안:

- “내 아이디어가 어떤 결과물로 바뀌는지” 예시 카드 추가
- 특허 초안, BMC, IR, 보고서 샘플 미리보기 추가
- 무료 체험과 유료 패키지의 차이를 첫 화면 하단에 간단히 노출

### 5.2 패키지 페이지

현재 `Inventor Beta`, `Venture Builder`, `Institution` 3단 구조는 적절하다.

보완 제안:

- 각 플랜에 “추천 대상”을 더 구체화
- `Venture Builder`는 베타 신청 또는 상담 예약 CTA로 연결
- 기관 플랜은 학교/기관/캠프/대회 운영 시나리오를 카드로 추가
- 결제 전이라도 “제공 결과물” 목록을 더 시각화

### 5.3 대시보드

대시보드는 사용자가 돈을 계속 내게 만드는 핵심 화면이다.

보완 제안:

- 사업화 준비도 점수
- 다음 추천 액션
- 누락된 입력 항목
- 멘토 리뷰 대기 상태
- 생성된 문서 목록
- 유료 권한/사용량 상태

현재 대시보드는 학습/AI 활동 기록 성격이 강하므로, 발명 사업화 운영 화면으로 더 전환할 필요가 있다.

### 5.4 관리자 화면

관리자 화면은 상용 운영에 매우 중요하다.

보완 제안:

- 사용자 목록
- 프로젝트 목록
- 결제/플랜 상태
- 리뷰 요청 큐
- 민감 작업 감사 로그
- 문의/기관 도입 리드 관리

기관 도입을 목표로 한다면 관리자 화면 완성도가 구매 결정에 직접 영향을 준다.

## 6. 기능 보완 우선순위

### P0: 상용화 전 필수

1. `DATABASE_URL` 환경 변수 정리
2. 서버 실행 가능 상태 복구
3. `npm run check` 타입 오류 해결
4. 로그인/회원가입/로그아웃 실제 동작 검증
5. 프로젝트 저장/수정/조회 검증
6. 관리자 권한 검증
7. Playwright 브라우저 설치 및 화면 캡처 검증

### P1: 유료 전환 품질

1. 사업화 준비도 대시보드 강화
2. 결과물 샘플 미리보기
3. 멘토 리뷰 요청 흐름
4. 유료 패키지 권한 모델
5. 기관 문의/상담 신청 폼
6. 결제 전환 전 Toss 테스트 결제 플로우

### P2: 성장 기능

1. 발명 프로젝트 템플릿
2. 대회/정부지원 제출용 문서 자동화
3. 파일 업로드 및 문서 보관
4. 이메일 알림
5. 사용량/성과 분석
6. 기관별 결과 리포트

## 7. 추천 작업 순서

1. 환경 변수와 DB 연결부터 복구한다.
2. 서버 타입 오류를 해결해 `npm run check`를 통과시킨다.
3. 홈과 패키지 페이지를 실제 브라우저로 캡처해 모바일/데스크톱 레이아웃을 검수한다.
4. 대시보드를 “학습 현황”에서 “사업화 운영 현황” 중심으로 개편한다.
5. 기관 도입 문의 폼과 관리자 리드 관리 화면을 연결한다.
6. 결제는 바로 활성화하지 말고 Toss 테스트 결제와 권한 부여까지 검증한 뒤 연다.

## 8. 결론

현재 사이트는 상용화 메시지와 패키지 구조 면에서 좋은 방향으로 개선되어 있다. 특히 “발명 아이디어 → IP 준비 → 시장 검증 → 사업화 자료 → 멘토/기관 운영” 흐름은 사용자가 비용을 지불할 이유를 만들 수 있는 구조다.

하지만 운영형 서비스로 공개하기에는 아직 기술 검증이 부족하다. 가장 중요한 남은 작업은 DB 연결, 서버 타입 오류 해결, 실제 브라우저 QA, 대시보드/관리자 기능 완성이다.

최종 의견:

지금 상태는 “상용화 랜딩 및 패키지 제안 초안으로는 활용 가능”하다.  
실제 유료 결제와 기관 도입 제안 전에는 P0 항목을 먼저 완료해야 한다.
